# Career FIT OS — 변경 이력

> 최초 기획서(`career_fit_prompt.md`) 대비 현재 구현까지의 변경사항.
> 1차 출처: 3주차 미션 회고(`3주차 미션 Career FIT OS.md`, 최지예 작성).

---

## 변하지 않은 핵심 (정체성)

- **철학** — 나도 회사를 고른다 · 서로 맞아야 오래간다 · 최종 결정은 내가 한다
- **패스해도 지원 가능** — 판정이 부정적이어도 사용자가 진행 선택 가능
- **이력서 1회 입력 → 이후 공고만** — 온보딩 1회, 반복은 공고만
- **각자의 기준** — 판단 기준(직무·환경·성장)은 사용자별로 온보딩에서 생성·수정 (2026-05 이후. 그 전엔 제작자 1인용·K-뷰티 기준 고정이었음)

## 주요 변경사항 (시간순)

### 1. 첫 구현 — 전부 "보류" 판정 문제
공고를 넣으면 다 "보류"가 떴음. 원인은 판단 기준이 틀렸던 것.
"브랜드 기획 직무냐"로 잡혀 있었는데 본인 경력과 안 맞는 기준이었음.
→ 더 근본 문제 발견: 판단 도구 전에 판단 기준이 먼저였다.

### 2. 기준 재설정 — 본인 분석부터 다시
해외영업 6년 + IT 마케터 1년, 화장품 경력 없음을 기준으로
온보딩 판단 기준을 4개로 교체.

### 3. 핵심 기준 4개 → 3개 통합
실제로 쓰면서 직무 적합 / 환경 적합 / 성장 적합 3개로 통합·정교화.
(기존 2·4번이 "브랜드 전체 흐름"으로 중복됐던 것을 해소)
네이밍도 "4대 기준 체크" → "핵심 기준 체크".

### 4. 화장품 경력 필수 감지 추가
화장품 경력이 없으니 JD에 "화장품 경력 필수" 문구 있으면 자동 경고.
처음엔 뺄까 고민했으나 — 지원 못 할 공고를 미리 걸러주는 시간 절약 기능으로 판단.

### 5. 이력서 초안 자동 생성 추가
공고 판단만으로 부족. 괜찮은 공고 찾아도 이력서 맞추는 걸 매번 새로 고민해야 했음.
JD 기반으로 자기소개 / 핵심 경력 3줄 / 지원 동기 초안 자동 생성. 복붙 가능 수준.

### 6. 구조 개편 — 2탭 → 3탭
처음엔 온보딩 / JD 분석기 2탭. 분석 기록이 섞여 복잡했음.
프로필 / 공고 판단 / 지원 현황 3개로 분리하면서 흐름이 명확해짐.

### 7. 다크모드 전환
밝은 베이지 톤(#F8F5EC)에서 판정 배지 구분이 약했음.
어둡게 바꾸니 지원 추천(초록) / 검토 필요(노랑) / 패스(회색)가 확실히 구분됨.

### 8. FIT 점수 체계 도입
정량 점수 추가. (초기엔 전부 100으로 나와 변별력 부족 → 재설계 중)

### 9. 최소 멀티유저 — 기준 개인화 (2026-05)
제작자 1인용·K-뷰티 하드코딩을 걷어내고, 사용자가 자기 기준으로 자기 직무 공고를 판정받게 함.
- **CONFIG·프롬프트 범용화** — `candidate.name/profile`, K-뷰티 페르소나·어필·목표 하드코딩 제거. CONFIG는 범용 폴백 템플릿으로만 남김. (커밋 `28e3a18`)
- **온보딩이 본인 기준 추출** — 이력서에서 커리어 목표(goal) + 3축 판단 기준 정의(직무/환경/성장 def·basis) + 어필 우선순위를 뽑아 프로필에 저장. 결과 화면에서 직접 수정 가능. (커밋 `b62cd7b`)
- **JD 분석이 프로필 기준으로 판정** — 분석 프롬프트가 프로필의 criteria·goal·어필 우선순위를 우선 적용(없으면 CONFIG 폴백). 3축 key/label/비중은 고정 → 판정·보드 로직 안정성 유지. (커밋 `5ea5626`)
- **필수 요건 필터 산업 중립화** — "화장품 경력 필수 감지"를 [내 프로필] 대비 JD 필수 요건 충족 여부(`requirement: mismatch|ok`)로 일반화. 레거시 `cosmetics` 결과도 호환 표시. (커밋 `2069d74`)
- 데이터 분리/동기화는 기존 localStorage 방식 그대로 유지(브라우저별 분리, CLOUD_SYNC=false). 기존 프로필은 재온보딩 전까지 범용 기준으로 폴백.
