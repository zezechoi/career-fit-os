## [2026-05-27] 최소 멀티유저 구현 — 기준 개인화 방식

- **결정:** "풀 멀티유저"가 아니라 딱 "사용자가 자기 기준으로 자기 직무 공고를 판정받는 것"까지만 구현. 4단계로 나눠 각 단계를 작동 상태로 커밋.
  1. CONFIG·프롬프트에서 제작자/K-뷰티 하드코딩 제거 → 범용 폴백 템플릿으로 (`28e3a18`)
  2. 온보딩이 이력서에서 본인 기준 추출(goal·3축 def/basis·어필 우선순위) → 프로필 저장, 결과 화면에서 수정 가능 (`b62cd7b`)
  3. JD 분석이 프로필 기준 우선 적용(폴백 CONFIG), 프로필 goal 주입 (`5ea5626`)
  4. 필수 요건 필터 산업 중립화: `cosmetics` → `requirement{level:mismatch|ok}` ([내 프로필] 대비 판정) (`2069d74`)
- **핵심 설계:** 3축 **key(job/env/growth)·라벨·비중은 고정**, **의미(def/basis)만 개인화**. 키를 고정해야 판정·보드·점수 로직이 안정적이고, 의미만 바꿔도 사용자별 판정이 된다.
- **확정한 선택지:**
  - ① 제작자(최지예)의 기존 프로필은 개인화 기준이 없으므로 **재온보딩 전까지 범용 기준으로 폴백**. 마이그레이션 코드 없이 폴백으로 흡수.
  - ② 추출된 기준은 **온보딩 결과 화면에서 직접 수정 가능**(goal·3축 def·어필 우선순위 전부 editable).
- **건드리지 않은 것:** 데이터 분리/동기화는 기존 localStorage 방식 그대로(브라우저별 분리, CLOUD_SYNC=false). 결제·권한관리 등 과한 기능 안 만듦.
- **레거시 호환:** 기존 히스토리의 `cosmetics` 결과는 표시 단계에서 `mismatch/ok`로 매핑해 그대로 보여줌(`required`→경고).
- **대안:** SYSTEM_PROMPT를 프로필별 함수로 만드는 안 → 기각. 시스템 프롬프트는 범용 역할 정의만 두고, 개인 컨텍스트(프로필·goal)는 분석 시점 user 메시지(`buildProfileText`)로 주입하는 게 더 단순·안정적.

## [2026-05] 차별점 + 타깃 정의 (경쟁 리서치 기반)

### 차별점 — 시장의 다른 도구와 무엇이 다른가
경쟁 리서치 결과, 원티드·리멤버·사람인·AI Resume Screener 등 시장의 거의 모든
도구가 **"기업이 사람을 고르는"** 방향(JD 보고 인재 추천, 합격 확률 예측)이었음.
Career FIT OS는 정반대 — **"내가 회사를 고르는"** 도구.

핵심 차별점 3가지:
1. **선택하는 도구** (vs 선택받는 도구) — 합격 확률이 아니라 "나랑 맞는가" 적합도 판단. 시장 전체와 관점이 반대라 모방이 어렵다.
2. **플랫폼 독립** — 원티드는 원티드 공고만, 사람인은 사람인 공고만 본다. 우리는 이미지/텍스트로 **어떤 플랫폼의 공고든** 한 기준으로 판단. 플랫폼 사업자는 자기 공고를 밀어야 해서 구조적으로 못 하는 영역.
3. **내 기준 설정** — 합격 확률은 플랫폼 알고리즘이 정하지만, 우리는 사용자가 직무·환경·성장 기준을 직접 세팅.

> 주의: "예상 면접 질문"은 원티드에 이미 있음(AI 피드백까지). 데이터 양에서 못 이기는 싸움 → 차별점이 아니므로 우선순위 낮춤.

> 정직한 한계: JD만으로 회사 분위기를 100% 알 수는 없음. "정확한 예측"이 아니라 "지원 전 거르개(업무+분위기 방향성 판단)"로 포지셔닝.

### 타깃 — 누구를 위한 도구인가
**"경력은 있는데, 다음 자리는 신중하게 고르고 싶은 사람."**
이직자 · 중고신입 · 공백기 후 복귀자. 공통 심리 = "또 잘못 가면 안 된다."
(신입의 "일단 붙자"와 반대. 합격 확률보다 적합도가 중요한 층.)

**시장 근거 (리서치 확인):**
- 인크루트 2026 HR 조사(인사담당자 650명): "더 강화된 중고신입 선호"가 **이슈 1위(33.5%)**. AI 일자리 감소(21.5%)보다 높음.
- 기업 77%가 신입공채에 중고신입 지원받음 / 인사담당자 86.2% 중고신입 채용 의사.
- 채용 명시 인원: 2023년 246만 → 2024년 192만 → 2025년 154만 명(2년간 약 100만 감소).
- 신입 채용 축소·경력 선호는 일시적 흐름이 아니라 "구조적 변화"로 분석됨.

**검증 상태:**
- ✅ 확인됨: 중고신입·경력직 선호 시장이 크고 커지는 중 (위 데이터).
- ⏳ 미검증: 이 층이 "적합도 판단 도구"를 실제로 원하는가. → 목요일 유저 2명 피드백으로 확인.
- 구분: 시장 존재(데이터 확인) ≠ 제품 수요(검증 필요). 발표 시 이 선을 지킬 것.

## [2026-05] 확장 시 비용 구조 — 각자 API 키 발급 방향 (미래 검토)

- **결정:** 아직 결정 아님. 사용자가 늘 때를 대비한 방향 메모.
- **배경:** 현재 2명 테스트는 "내 키 공유 + 비밀번호" 방식. 편의 우선이라 소수까지만 유효. 사람이 늘면 (1) 내 비용 부담이 커지고 (2) 비번 유출 시 통제 불가 → 지속 불가능.
- **방향 후보:**
  - **A. 각자 키 발급** — 사용자가 본인 API 키 입력, 자기 쓴 만큼 자기 부담. 내 비용 0. (피드백에서 제안받은 방향.) 단점: 결제·발급 허들로 비개발자 이탈.
  - **B. 내가 전액 부담** — 사용자는 편하나 사람 늘수록 내 지갑 부담. 무료 배포 아니면 불가.
  - **C. 내가 호출 + 사용자 과금(SaaS)** — 구독·사용료로 API 비용 충당. 정석이나 결제·사용량측정·회원관리 필요. 제일 무거움.
- **단계 가이드:** 지금(2명)=내 키 공유 / 무료로 더 열 때=A(각자 키) / 제품화·수익화=C.
- **트리거:** 2명 검증이 "계속 쓰겠다"로 나오면 그때 A vs C 결정.

## [2026-05] 1인용 → 소규모 멀티유저로 전환

- **결정:** 본인 1인용에서, 몇 명이 함께 테스트하는 멀티유저 구조로 확장. 검증되면 더 넓게 열 수 있음.
- **이유:** MVP가 셀피쉬클럽에서 선정되어 발표함. "남들도 써보고 싶다"는 반응으로 수요 신호 확인. 한 번에 제품화하지 않고 소규모 테스트부터 단계적으로 연다.
- **방식:** API 키는 사용자가 각자 입력 (서비스가 비용 부담·결제 시스템 만들지 않음). 판단 기준 3개는 고정에서 사용자별 설정 가능한 템플릿으로. 데이터는 사용자별 분리.
- **주의:** 아래 "범용 → 1인용으로 좁힘" 결정을 되돌리는 것. 모순이 아니라 — 처음엔 빠른 완성을 위해 의도적으로 좁혔고, 검증된 뒤 다시 연 흐름이다.

## [2026-05] 범용 서비스 → 1인용으로 좁힘

- **결정:** 다른 사람도 쓸 수 있는 범용 서비스 대신 나만 쓰는 툴로.
- **이유:** 범용으로 만들려다 막혔음. API 공유 방법까지 고민했으나, 나만 쓰는 툴로 방향 잡으니 오히려 빠르게 완성됨. 고객이 한 명이면 판단 기준·기능·사용 이유가 전부 명확해진다.
- **교훈:** 가장 단순한 게 이긴다.

## [2026-05] 핵심 기준 체크 4개 → 3개 통합

- **결정:** 직무 적합 / 환경 적합 / 성장 적합 3개로 정리.
- **이유:** 기존 2번(브랜드 흐름을 볼 수 있는 포지션)과 4번(브랜드 흐름을 경험할 수 있는 환경)이 의미상 거의 동일. 실제로 쓰면서 중복을 발견해 통합·정교화. 기준이 적을수록 지원/패스 판단이 단순.
- **대안:** 4개 유지(도메인 적합성 분리). 도메인은 "직무 적합"에 흡수.

## [2026-05] 화장품 경력 필수 감지를 빼지 않고 살림

- **결정:** JD에 "화장품 경력 필수"가 있으면 자동 경고하는 기능 유지.
- **이유:** 화장품 경력이 없어서 처음엔 이 기능을 뺄까 고민. 하지만 지원할 수 없는 공고를 미리 걸러주는 = 시간을 아껴주는 기능이라고 판단. 거르는 것도 OS의 역할.
- **주의:** 1인용 시절 본인 맞춤 기능. 멀티유저로 가면서 산업 중립적으로 일반화하거나, 사용자가 켜고 끌 수 있는 옵션으로 전환 필요.
